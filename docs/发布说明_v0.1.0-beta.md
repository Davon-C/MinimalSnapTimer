# 发布说明 v0.1.0-beta

## 当前发布策略

- 仅提供完整版单文件 exe
- 不再提供 lite 包
- 默认面向普通用户直接分发

发布文件：

- `MinimalSnapTimer_v0.1.0-beta_win-x64.exe`

## 本轮重点改动

- 修复从托盘菜单打开设置时，主窗口隐藏或未显示导致设置窗口 Owner 设置失败的问题
- 设置窗口现在会在无可用 Owner 时居中显示，不再弹出 `.NET` 未处理异常窗口
- 完成一轮常驻资源轻量化审计，确认当前内存占用总体合理
- 托盘状态刷新改为增量更新，避免计时过程中每秒重建整套托盘菜单
- 普通日志增加滚动与数量限制，避免长期运行导致日志目录持续膨胀
- 通知服务改为懒创建 `MediaPlayer`，降低未播放自定义声音时的常驻对象占用
- 保持现有倒计时、纯时间模式、点击穿透、托盘恢复、多显示器、提醒链路和启动保护逻辑不回退

## 为什么只保留完整版

原因很直接：

- 普通用户无需安装 `.NET Desktop Runtime`
- 分发更简单
- 环境差异更少
- 手工验收路径更稳定

## 为什么 exe 体积较大

因为当前发布的是：

- `.NET 8`
- `WPF`
- `win-x64`
- `self-contained`
- `single-file`

运行时和 WPF 依赖会一起打包进 exe，所以体积较大属于正常现象。

## 当前内存占用说明

基于本轮发布版实测：

- 冷启动后 10 秒，`Private Memory` 大约在 `57 MB` 左右
- 空闲 1 分钟后，`Private Memory` 仍大致维持在 `57 MB` 左右
- 进入纯时间模式、最小化到托盘、完成一次 1 分钟倒计时后，私有内存没有出现持续上涨趋势
- 打开设置窗口会短时拉高内存，但关闭后会回落一部分，这更像 WPF 缓存和对象保留，不等于泄漏

结论：

- 对 `WPF + self-contained + 托盘 + 多语言 + 主题 + 通知` 这类桌面工具来说，这一档内存占用是合理的
- 当前阶段不建议因为 `48.7 MB` 左右的常驻内存去做从 0 重写

## 不建议从 0 重写的原因

- 当前功能链路已经稳定
- 已经补齐了启动保护、设置回退、提醒链路和托盘恢复
- 真实常驻内存没有表现出“必须重写才能解决”的级别
- 重写成本远高于当前可获得的收益

如果未来目标从“可用、稳定、轻量”升级为“极致轻量”，那时再评估原生技术路线更合理。

## 当前已知限制

- 跟随系统主题不保证实时监听系统变化
- English 下少量边角文本仍可能未完全覆盖
- 原生 Toast 仍是轻量实现
